# Label Conflict Resolution Tool

## Overview

`resolve_label_conflicts.py` automatically detects and resolves label conflicts in similar image groups within dataset annotation files.

**Purpose**: Identify images with:
- High visual similarity (≥ 0.9 perceptual hash similarity)
- High ROI (Region of Interest) overlap (IoU ≥ 0.7)
- **Different labels** for the same physical region

Then enable batch selection and automated label standardization.

## Features

- **Automatic Detection**: Scans similarity groups and identifies label inconsistencies
- **Batch Processing**: Select a master label once; apply to entire conflicting group
- **Non-destructive**: Original files backed up before any updates
- **Comprehensive Reports**: Generates JSON reports and human-readable summaries
- **Two Modes**: Interactive selection (`--auto` for batch) or manual choice per conflict

## Installation

Requires Python 3.7+ with standard libraries (json, pathlib, argparse). No external dependencies.

## Usage

### Basic Auto Mode (Non-interactive)
```bash
python resolve_label_conflicts.py --folder auto_labels_preview_screen_20260515_161232 --auto
```
Automatically selects the first label for each conflict and updates all others to match.

### Interactive Mode (Default)
```bash
python resolve_label_conflicts.py --folder auto_labels_preview_screen_20260515_161232
```
Lists all detected conflicts and prompts you to choose a master label for each.

### Custom Thresholds
```bash
python resolve_label_conflicts.py \
  --folder auto_labels_preview_screen_20260515_161232 \
  --similarity-threshold 0.85 \
  --iou-threshold 0.6 \
  --auto
```

### Full Command Reference
```
--folder FOLDER_NAME                  Required. Name of the preview folder (e.g., auto_labels_preview_screen_20260515_161232)
--recordings-dir PATH                 Optional. Path to recordings directory (default: recordings)
--similarity-threshold FLOAT           Optional. Min similarity to consider (default: 0.9)
--iou-threshold FLOAT                 Optional. Min IoU for ROI overlap (default: 0.7)
--auto                                Optional. Auto-select first label (non-interactive mode)
```

## Input Requirements

Script expects the following structure:
```
recordings/
  {folder_name}/
    annotations_labelme/
      event_0001_*.json
      event_0002_*.json
      ...
    reports/
      similarity_groups.json
```

- **similarity_groups.json**: Generated by the auto-labeling pipeline; contains cluster info and member relationships
- **annotations_labelme/**: LabelMe-format JSON files with shape labels and coordinates

## Output

The script generates three output files in the `recordings/` directory:

### 1. `{folder_name}_conflict_report.json`
Detailed conflict detection results:
- Total conflicts found
- Per-conflict details: cluster ID, members, labels, ROI coordinates, IoU values
- Unique labels involved in conflicts

### 2. `{folder_name}_resolution_log.json`
Update tracking information:
- Timestamp of operation
- Number of conflicts resolved
- List of files updated with new labels
- Failed updates (if any) with reasons

### 3. `{folder_name}_resolution_summary.txt`
Human-readable summary with:
- Folder and timestamp info
- Detection and resolution statistics
- List of updated files with new labels
- Backup location for original files

### 4. Backup Directory
```
{folder_name}/_backup/YYYYMMDD_HHMMSS/
  original_annotation_files.json
```
All original annotation files are preserved before any modifications.

## Workflow

### Phase 1: Load & Validate
- Verify folder exists and `similarity_groups.json` is present
- Load similarity group clustering data

### Phase 2: Detect Conflicts
- Iterate through all cluster members
- For each pair with similarity ≥ 0.9:
  - Load annotation files
  - Compute shape-level ROI overlap (IoU)
  - Check if overlapping shapes have different labels
  - Record conflict details

### Phase 3: Interactive Selection (Optional)
- Display all conflicts with candidate labels
- Accept user input: `conflict_idx:label_choice` format
- Example: `0:CHT帳號  1:Login button  2:CHT帳號`
- Or auto-select first label for each conflict with `--auto`

### Phase 4: Apply Updates
- For each selected conflict, update the non-selected member's annotations
- Backup original files to `_backup/{timestamp}/`
- Write updated LabelMe JSON with new labels

### Phase 5: Generate Reports
- Create detailed conflict report (JSON)
- Create resolution log (JSON)
- Generate summary (human-readable TXT)

## Troubleshooting

### No conflicts found
- Verify `similarity_groups.json` exists and is valid
- Check that cluster members have similar visual content
- Verify member annotations exist in `annotations_labelme/`

### Updates fail with "no matching shape"
- May indicate shape coordinates don't overlap at IoU ≥ 0.5 threshold
- Try lower IoU threshold: `--iou-threshold 0.3`

### Encoding issues in output
- Use UTF-8 compatible terminal/IDE
- Reports are saved as UTF-8; display issues are terminal-dependent

## Performance Notes

- For folders with < 100 conflicts: typically < 5 seconds
- Path resolution uses forward slashes internally; handles Windows backslashes automatically
- Shape matching uses normalized IoU (Intersection over Union) calculation

## Best Practices

1. **Review before update**: Always check the `_conflict_report.json` first
2. **Backup first**: Script auto-backs up, but keep additional copies of `annotations_labelme/`
3. **Test incrementally**: Try `--auto` on smaller datasets first
4. **Validate labels**: After updates, spot-check several resolved conflicts
5. **Version control**: Commit results to git after successful resolution
